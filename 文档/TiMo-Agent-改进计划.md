## TiMo 人设定义

**混合模式**：以温柔鼓励为基础底色，用户懈怠时加入严格督促，数据分析时切换专业口吻。始终保持善意和支持感，绝不冒犯用户。

| 场景 | 风格 | 示例语气 |
|------|------|----------|
| 日常对话 | 温柔鼓励 | "没关系，慢慢来，每天进步一点点" |
| 用户懈怠/断签 | 严格督促 | "今天还没打卡哦，别忘了你的目标" |
| 数据分析/周报 | 专业导师 | "本周正确率 72%，比上周提升 8%" |
| 首次对话 | 亲切自然 | 询问昵称偏好，建立关系 |

## Phase 1：短文生成修复（硬伤，必须可用）

### 问题
- AI 把单词硬塞进句子，语法完全错误（如 "His ambiguous to learn..."）
- 降级 fallback 只是把模板句子拼接，不是完整短文

### 改进方案
1. **Prompt 重写**：要求 AI 先构建故事线，再自然融入目标词
   - 用英文 prompt（DeepSeek 对英文指令理解更好）
   - 明确要求：先有故事，再嵌入单词，不能硬塞
   - 单词保持原形，不改变词性（不加 -s, -ed, -ing）

2. **质量校验**：生成后自动检查
   - 每个目标单词必须出现在短文中
   - 短文长度 120-180 词
   - 不达标时自动重试一次（更严格的 prompt）

3. **降级增强**：API 失败时不再返回乱造句
   - 改为返回预设的高质量模板短文
   - 按单词数量选择不同模板

4. **新增字段**：
   - `title`：短文标题
   - `translation`：中文翻译摘要（2-3 句）
   - `wordSentences`：每个单词的独立例句保留

5. **难度选项**：根据用户水平自动选择初级/中级（后续可扩展）

### 技术改动
- `AgentService.generatePassage()`：重写 prompt + 质量校验 + 重试逻辑
- `AgentService.buildFallbackPassage()`：改进 fallback 短文质量
- `PassageResponse`：新增 `title`、`translation` 字段

### 验收标准
- 返回的短文必须是完整、有意义的故事或段落，不是拼接句子
- 短文长度控制在 120-180 词
- 所有目标词都必须以原形出现在短文中
- 每个目标词都保留单独例句
- DeepSeek 不可用时，fallback 也必须可读、自然、可演示

## Phase 2：推荐逻辑智能化

### 问题
- 0 学习量也建议 30 新词，不看用户实际水平
- 推荐模式固定，不考虑用户是否有顽固词

### 改进方案
1. **冷启动推荐**：
   - 新用户首日 5-10 词，逐步递增
   - 第 1 天：5-10 词
   - 第 2-3 天：10-15 词
   - 第 4-7 天：15-20 词
   - 7 天后：根据表现调整到 20-30 词

2. **正确率动态调整**：
   - < 60%：减少新词量到 5-10
   - 60-85%：保持当前推荐量
   - > 85%：可适当增加到 25-30

3. **打卡天数影响**：
   - 连续打卡 ≥ 3 天：可适当增加
   - 断签后恢复：回到基础量

4. **模式推荐智能**：
   - 有 ≥ 5 个顽固词 → 优先推荐 `context_deep`
   - 有待复习词 > 新词 → 优先推荐 `unified_review`
   - 新用户无数据 → 推荐 `quick_memory`

### 技术改动
- `AgentService.getRecommendation()`：重写推荐逻辑
- 新增冷启动判断、正确率分析、顽固词优先级

### 验收标准
- 新用户不会直接拿到 30 个新词
- 推荐量会随学习表现变化
- 有顽固词时，优先推荐深度学习或复习
- 推荐理由必须和用户真实数据一致

## Phase 3：TiMo 对话体验升级（核心差异）

### 问题
- 每轮都在推功能，像推销不像教练
- 状态标记混乱（正常聊天也标 `alert`）
- 用邮箱前缀 "fullstack" 称呼用户
- 缺乏意图识别，所有回复一个模板

### 改进方案

#### 3.1 人设系统提示词重写
```
你是 TiMo，一个温柔但专业的备考教练。你的风格：
- 日常对话：温柔鼓励，像朋友一样支持用户
- 用户懈怠时：适当严格，提醒目标但不冒犯
- 数据分析时：专业客观，用数据说话
- 始终保持善意和支持感，绝不冒犯用户
```

#### 3.2 意图识别层
| 用户意图 | 识别关键词 | 回应策略 |
|----------|-----------|----------|
| 闲聊/问候 | 你好、在吗、hello | 友好回应，自然引导 |
| 问单词 | 怎么记、什么意思、abandon | 给出记忆法 + 例句 |
| 求建议 | 应该怎么学、推荐 | 基于数据给建议 |
| 抱怨/疲惫 | 好累、不想学、太难 | 共情 + 轻量建议 |
| 求鼓励 | 我进步了吗、怎么样 | 数据 + 真诚鼓励 |
| 设置昵称 | 叫我XX、昵称 | 存入用户偏好 |

#### 3.3 减少推销感
- 不每轮都推功能
- 根据对话上下文自然引导
- 3 轮对话内最多推 1 次功能
- 用户明确说不想学时，尊重并给轻量选项

#### 3.4 状态标记修复
| 场景 | 正确状态 |
|------|----------|
| 正常聊天 | `idle` |
| 完成学习/答对 | `success` |
| 落后进度/连续断签 | `alert` |
| API 不可用 | `offline` |

#### 3.5 昵称收集与称呼逻辑
- 首次对话：TiMo 问"你希望我怎么称呼你？"
- 存入 `user.nickname` 字段
- 有昵称 → 用昵称
- 无昵称 → 用"同学"
- 绝对不用邮箱前缀

### 技术改动
- `AgentService.sendChat()`：重写 system prompt + 意图识别
- `AgentController` / `UserService`：新增昵称设置接口
- 前端 `TiMoDialog.vue`：首次对话引导昵称设置
- 状态标记逻辑修正

### 验收标准
- 正常聊天默认是 `idle`，不会频繁误报 `alert`
- TiMo 说话要温柔、支持感强，不冒犯用户
- 用户懈怠时，能明显体现督促感，但不能阴阳怪气
- 数据分析时，TiMo 必须用专业导师口吻给出结论
- 首次对话能自然询问昵称偏好，并存储成功

## Phase 4：周报 / 数据洞察升级

### 问题
- 全是模板空话，没有数据洞察
- 无数据时回复干巴巴

### 改进方案
1. **有数据时**：
   - 具体趋势分析（"本周比上周多学了 50 个词"）
   - 正确率变化趋势
   - 顽固词减少情况
   - 具体可执行建议

2. **无数据时**：
   - 温暖引导，而不是干巴巴的"学了 0 个词"
   - 给出第一步建议（"先从 5 个词开始试试？"）

### 技术改动
- `AgentService.getWeeklyReport()`：重写数据聚合和 AI prompt
- `AgentService.generateSessionReport()`：改进 session 报告

### 验收标准
- 周报必须包含至少 1 条具体洞察，不只是模板句
- 有数据时要说明趋势，无数据时要给出第一步建议
- session report 要更像课后总结，而不是泛泛鼓励

## Phase 5：单词分析完善

### 问题
- 音标为空
- 记忆法偏套路

### 改进方案
1. 音标从词库获取，不依赖 AI 生成
2. 记忆法多样化：谐音、词根词缀、场景联想、拆字法
3. 增加同义词/反义词对比
4. 增加常见搭配和用法说明

### 技术改动
- `AgentService.analyzeWord()`：优先从 word 表获取音标
- 改进 AI prompt 要求多样化记忆法

### 验收标准
- 音标不应长期为空
- 记忆法不能只靠单一套路，要有词根、联想、语境等变化
- 至少能给出可直接讲给用户听的简洁解释

## 执行原则

1. 先保证答辩演示效果，再逐步优化边角功能。
2. 所有 AI 回复都先以“可读、自然、可解释”为最低门槛。
3. 任何新逻辑都必须通过现有后端和前端测试。
4. 用户体验优先，不能为了“智能”牺牲善意和清晰度。
5. 不把问题藏到 fallback 里，fallback 也要可演示。

## 风险与回滚

1. **短文生成质量不稳定**
   - 回滚：直接切到高质量 fallback 模板
   - 兜底：保留 `wordSentences`，保证前端可展示

2. **对话过于严格或过于热情**
   - 回滚：将 system prompt 收敛回温柔鼓励模式
   - 兜底：默认称呼改回“同学”

3. **推荐逻辑过于激进**
   - 回滚：恢复当前稳定阈值
   - 兜底：限制新词量上限，避免一次推太多

4. **昵称收集影响首次体验**
   - 回滚：允许跳过昵称设置
   - 兜底：未设置时统一使用“同学”
